# OGSM Framework — 中文使用指南

> 英文原版請參考同目錄的 [SKILL.md](./SKILL.md)（Claude agent 讀取用）

---

## 這個 Skill 的由來

這個 skill 的方法論來自**張敏敏老師**的 OGSM 系統，她是台灣最重要的 OGSM 實踐者，著有《OGSM 打造高敏捷團隊》與《OGSM 變革領導》（均由商業周刊出版）。

我們把她的框架移植到 AI agent 團隊的工作流程裡。原因很簡單：多個 agent 同時運作時，最容易出現的問題不是技術問題，而是**方向不一致**——每個 agent 都在做自己覺得正確的事，但合在一起卻沒有靈魂。OGSM 強迫你在每一層都回答「這跟最終目標的關係是什麼？」，讓整個 agent 團隊對齊在同一個方向上。

第一個實際應用的場景是 AIA CEU 建築師繼續教育課程的 18 個 agent 團隊，那個經驗讓我們確認 OGSM 在 AI agent 協作上是有效的。

---

## OGSM 到底是什麼（最重要的部分先說）

**OGSM 不是 OKR。** 這一點非常重要，如果你把它寫成 OKR 的樣子（任務清單加上達成檢查點），你就已經用錯了。

| 層次 | 它真正是什麼 | 常見錯誤 |
|------|-------------|---------|
| **O**（Objective，目標） | 最終目的地。要有情感，有方向感。 | 寫太虛（「成為最好的」） |
| **G**（Goal，目標成果） | 階段性的整合成果，**必須能連回 O**。每讀一個 G，都要能感受到目標對象的存在。 | 把任務清單寫在這裡 |
| **S**（Strategy，策略） | 針對「這群人」選擇的這條路。必須回答「為什麼走這條路，而不是另一條？」 | 寫出適用任何人的通用方法 |
| **M**（Measure，衡量） | 任務 + 驗收標準 + 品質關卡 | 沒寫，或跟 G 混在一起 |

### 黃金法則

**每個 G 讀起來都要能感受到目標對象的存在。**

如果你的 G 拿掉對象名稱還是成立，那它不是 G，是 M。

---

## OGSM vs OKR：為什麼要用 OGSM

- **OKR** = Google 工程師風格（務實、數字導向、適合追蹤進度）
- **OGSM** = 一般人的風格（故事導向、以對象為核心、適合對齊方向）

OKR 問的是「我們達成了多少？」
OGSM 問的是「我們為什麼對這群人這樣做？」

純任務清單會失去「為什麼」，讓整個團隊變成機械式執行。OGSM 強迫你在每一步都回答：這對**這群特定的人**，為什麼重要？

## 5 個反模式（你一定會踩到）

### 反模式 1：G 其實是 M

**症狀**：讀 G 的感覺像在讀任務清單。

**錯誤範例**：
> G：提供 5 個包含 3 個資料來源的案例研究

**正確範例**：
> G：建築師讀完這份材料後，在專案會議上可以直接引用正確的規範條文，不怕被糾正

**判斷方法**：問自己「這個 G 有提到目標對象嗎？」沒有的話，重寫。

---

### 反模式 2：S 是通用方法

**症狀**：這個策略換成任何專案、任何對象都可以用。

**錯誤範例**：
> S：用 markdown 格式，每 10 分鐘加一個互動檢查點

**正確範例**：
> S：使用 MasterFormat 章節編號和 AIA 合約語言——建築師認得這套參照系統，才會覺得有公信力

**判斷方法**：在每個策略後面試著加「針對 [對象]，因為 [原因]」。如果你加不上去，它不是策略，只是作法。

---

### 反模式 3：所有 reviewer 都是內部人

**症狀**：每個品質審查者都是公司內部角色。

**錯誤範例**：
> reviewer：內容主管、品質長、專案負責人

**正確範例**：
> reviewer：
> - 目標對象 persona（用不同的 AI 模型扮演目標對象，讀完後給回饋）
> - 相鄰角色（例如目標是建築師，加一個「承包商」或「業務員」視角）
> - 陌生眼睛（完全沒有專案背景的 reviewer，看是否能獨立理解）

**為什麼重要**：全部都是內部人會產生確認偏誤——每個人都覺得 OK，結果真正的目標對象看了卻不知所云。

建議：如果你的 agent 團隊用 Claude，就讓 Gemini 扮演目標對象來做最終審查，反之亦然。

---

### 反模式 4：O 從來沒有被直接驗證

**症狀**：最終關卡是技術性檢查，不是目標對象的真實體驗。

**錯誤範例**：
> 最終關卡：HTML 通過 W3C 驗證 / 所有連結有效 / 字數達標

**正確範例**：
> 最終關卡：讓 3 個 persona 模擬讀完後，能正確回答 5 道決策問題中的 4 道。未達標，不上線。

**關鍵**：你的 O 是關於人的感受或能力，你的最終關卡就必須測試那個感受或能力，不能用技術指標代替。

---

### 反模式 5：自己改自己的作業

**症狀**：團隊用自己的標準檢查自己的成果。

**錯誤範例**：
> 研究員寫完，研究員自己驗收。

**正確範例**：
> 引入外部標準：業界基準、目標對象 persona、不同的 AI 模型視角。

---

## 6 個工作流程步驟

### Step 1：定義 O（跟使用者一起）

O 要符合三個條件：
1. **有情感**（不只是功能性）——「讓建築師愛上這個學習材料」，而不只是「教建築師某件事」
2. **對象具體**——明確說出是誰
3. **成果導向**——描述對象能做到什麼、感受到什麼，不是我們產出了什麼

定義 O 前，問三個問題：
- 目標對象是誰？
- 完成後，他們應該處於什麼情緒狀態？
- 他們應該能夠獨立做到什麼？

**實際範例（AIA 建築師課程場景）：**
> O：讓建築師在讀完這份自動門 CEU 課程後，有信心在設計會議中主導自動門的規格討論，而不是把這個問題完全丟給廠商。

提出 O 草稿，確認後才進行下一步。

---

### Step 2：每個 Agent 各自寫自己的 G、S、M

把確認好的 O 同步給每個 agent，讓他們各自撰寫（可以並行）：

```markdown
### [角色名稱]

**G（階段整合成果——必須連回 O）**
> 一句話。讀起來要能感受到目標對象的存在。

**S（針對這群對象選擇的這條路）**
- 策略 1：具體方法 + 為什麼這對這群對象有效
- 策略 2：...
- 必須回答「為什麼不走另一條路？」
- 不接受通用方法。

**M（任務 + 驗收標準）**
- 量化目標（幾個、多長、哪些來源）
- 格式要求
- 驗收標準（誰來確認、怎麼確認）

**與 O 的連結**
一句話說明：為什麼達成這個 G，就推進了 O？
```

**實際範例（研究員 agent）：**

```markdown
### 研究員

**G**
> 讓建築師拿到的資料，是他們在設計會議現場可以直接引用的，
> 而不是還需要再去查原始規範才能用。

**S**
- 所有條文引用都附上 ANSI/BHMA 和 ADA 的原始章節號碼，
  因為建築師在會議上說「根據 ANSI A156.19」比說「根據某個標準」更有說服力
- 用 AIA 規格書的 MasterFormat 格式整理，
  因為建築師認得這套系統，會覺得是可以直接放進 spec 的材料

**M**
- 涵蓋自動門的 5 大規範（ANSI、ADA、IBC、NFPA、local amendment）
- 每個規範附上正式引用格式，可直接複製貼上
- 由目標對象 persona reviewer 驗證：能在 2 分鐘內找到所需條文

**與 O 的連結**
建築師有了可以直接引用的資料，才能「主導規格討論而不是把問題丟給廠商」。
```

---

### Step 3：加入衡量層（最少 3 個 agents）

這些 agent 跟「reviewer」不同——reviewer 查品質，衡量層追進度。

1. **Performance Supervisor**：監控每個 agent 的 G 達成率
2. **Quality Auditor**：驗證 S 的承諾有沒有被兌現
3. **Learning Outcome Validator**：直接測試 O 有沒有被達成（用 persona simulation）

---

### Step 4：加入外部 Reviewer（最重要的升級）

至少加 2 個外部視角的 reviewer：

- **目標對象 Persona Reviewer**：用不同的 AI 模型（比如團隊用 Claude，就用 Gemini 來扮演）模擬目標對象第一次讀到這份材料
- **相鄰角色 Reviewer**：有不同利益關係的人（比如目標是建築師，加「承包商」或「設施管理員」視角）
- **陌生眼睛 Reviewer**：完全沒有專案背景的 reviewer，測試材料能否獨立存在

**核心原則**：reviewer 不能全部都是內部公司角色。要用不同的 AI 模型和不同的 persona 視角。

---

### Step 5：對齊驗證矩陣

建立一張表：

| Agent | 主要 G | G 如何連結到 O | 如果 G 失敗，O 的風險 |
|-------|--------|---------------|----------------------|
| 研究員 | 可直接引用的規範資料 | 有資料才能主導討論 | 建築師沒有說服力，退回依賴廠商 |
| 課程設計師 | ... | ... | ... |

任何一行的「G 如何連結到 O」欄位如果很薄弱，那個 G 就是錯的，要重寫。

---

### Step 6：Wave Gates（關卡設計）

定義每個階段之間的關卡。最重要的原則：**最終關卡必須直接驗證 O**，不能只是確認任務完成。

**實際範例（建築師課程場景）：**

> 關卡 3（上線前）：Learning Outcome Validator 讓 3 個 persona 模擬完整閱讀，並回答 5 個設計決策問題。4/5 答對才算通過。未達標，返回修改，不上線。

這個關卡直接測試「建築師有沒有辦法在設計會議中主導討論」，而不是測試「HTML 有沒有錯誤」。

---

## 從真實專案學到的原則

這四條原則是在 HSW-002 第三次迭代時才真正學到的。前面兩次都因為沒有遵守這些原則而做錯方向。

### 原則 1：受眾的 workflow 理解

定義「受眾是誰」不夠——你還要搞清楚他們**實際在做什麼**。

- ❌ 錯誤：假設受眾會做你想教的事
- ✅ 正確：研究受眾的實際工作流程，然後定位你的內容

**實際翻車案例（HSW-002 v2 → v3 的迭代）：**

我們以為建築師會「寫 spec」，所以課程教了 spec 語言、MasterFormat 章節格式。錯了。

- 建築師**不寫 spec**——他們指定品牌，spec writer 處理細節
- 課程應該教「情境辨識」和「如何跟 spec writer 協作」
- 不是教「如何寫 spec」

**判斷方法**：問自己「這個受眾**實際**會做這件事嗎？」不確定的話，訪問一個真實的受眾。

---

### 原則 2：挑戰者 vs 既得利益者的敘事框架

如果你代表的是產業裡的挑戰者（不是既得利益者），你的內容要**賦能**受眾，不是把他們困在既得者的生態系。

**實際翻車案例（HSW-002 v3）：**

- 三大門五金廠（Allegion、ASSA ABLOY、dormakaba）控制了大部分 spec writer
- 建築師想用獨立廠商時，常常不知道去哪裡找資源
- 獨立廠商的課程責任：告訴建築師「你有其他選擇」
- 但要用**中立角度**：「這裡有獨立的資源路徑」，不是「用我們」

**判斷方法**：問自己「這個內容讀完，受眾更自由還是更依賴？」

- 更依賴 → 你在寫銷售簡報
- 更自由 → 你在做真正的教育

---

### 原則 3：資源路由就是交付物

有時候課程最好的產出不是「教他們怎麼做」，而是「告訴他們去哪裡」。

**實際案例（HSW-002 v3）：**

建築師不需要學會寫 spec。但他們需要知道：

- 遇到這種情境 → 這是 Waterson 情境
- 想用 Waterson → 這裡有 SpecLink、SPC Alliance 可以協助
- 三大廠以外還有其他資源

「資源路由」本身就是一個有價值的交付物，不是「沒教好」的代替品。

---

### 原則 4：第三方中立的框架

即使你從中受益，介紹資源時要用**中立立場**。

**範例**：

- ❌ 「用 Waterson！去 SpecLink 找我們！」
- ✅ 「獨立廠商的 spec 資源不只一條路。SpecLink、SPC Alliance、CSC/CSI 都是選項。」

**判斷方法**：讓 **Sales Rep Advisor**（外部 reviewer）讀一遍。如果他說「這裡聞起來像廣告」，就不夠中立。

---

## 反模式 6：把受眾的工作誤解了

前面列了 5 個反模式，這是從 HSW-002 v3 迭代加進來的第 6 個：

**症狀**：課程教的技能，受眾實際工作中不會用到。

**錯誤範例**：
> 教建築師怎麼寫 spec 的完整語法

**正確範例**：
> 教建築師怎麼辨識「這是 Waterson 的情境」+ 怎麼跟 spec writer 協作

**修復**：研究受眾真實工作流程。如果他們不做你教的事，改教他們真正做的事——可能是「路由到正確資源」，而不是「自己做」。

---

## M 必須對準 S 的資源承諾（加強說明）

這是第二版修訂才補上的關鍵原則。

**M 不是交付物清單，是資源使用驗證。**

**錯誤的 M：**
- S：以 ICC Digital Codes 作為主要資料來源
- M：記錄 4 條法規條文 ← 這是交付物，不是衡量

**正確的 M：**
- S：用 ICC Digital Codes 和 NFPA 官方出版品交叉驗證
- M：
  - 每條引用都經過兩個來源核對？
  - 交叉驗證修正了幾條錯誤？
  - 有沒有任一條只依賴單一來源？

**測試方法**：M 讀起來應該是「你的團隊有沒有真的用上 S 承諾的資源？」——不是「團隊產出了幾個東西？」

## 如何在你的專案裡使用這個 Skill

### 方法一：直接呼叫

在 Claude Code 裡輸入：

```
/ogsm-framework
```

Claude 會自動進入 OGSM 規劃流程，從問你 O 開始。

### 方法二：在規劃 agent 團隊時使用

每次用 `/agent-builder` 新增 agent 角色時，問自己：

> 這個 agent 的 G 是什麼？它怎麼連回整個專案的 O？

如果你說不清楚，這個角色的定義就還不夠清晰。

### 方法三：跟其他 skills 組合使用

| 組合 | 場景 |
|------|------|
| `/ogsm-framework` + `/aia-rewrite` | AIA CEU 課程的多 agent 團隊規劃 |
| `/ogsm-framework` + `/publish-article` | 部落格文章寫作團隊的目標對齊 |
| `/ogsm-framework` + `/agent-builder` | 新增 agent 時確認角色定位 |

### 常見使用流程

1. 新專案開始，先跑 `/ogsm-framework` 定義 O
2. 確認 O 後，派每個 agent 各自寫 G/S/M
3. 用對齊驗證矩陣確認所有 G 都連回 O
4. 設計關卡，最終關卡直接測試 O
5. 加入外部 reviewer（用不同 AI 模型）
